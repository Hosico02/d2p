## Observations

Three distinct signals separated, in order of how load-bearing each is for the next move:

### 1. The Verifier's reasoning is actually good. The parser is throwing it away.

Every row reports `categories: none`. That's not because the model failed
to enumerate findings — the raw response on `flask-bad-readme` contains
seven detailed `reasoning_trace` entries including the exact intended
catch ("README calls `pnpm test` but no package.json exists to define the
test script"). On `flask-clean` the raw response has six `findings`
entries pointing at the `ModuleNotFoundError: No module named 'app'`
collection failure (which is a real bug, see #2 below).

Yet `VerifyResult.new_finding_categories == []` and
`VerifyResult.reasoning_trace == []` after parsing.

**Hypothesis:** the minimax model emits the findings under a key that
`Verifier._parse_result` doesn't read (e.g., `findings` instead of
`new_finding_categories`), and/or emits per-finding shape that doesn't
match the `Finding` dataclass schema. The verdict (`fail`/`needs_repair`)
makes it through; everything else doesn't. The `blocking_findings: 2x
unspecified empty stubs` on flask-clean smells like a fallback insertion
in the parser when it sees a `fail` verdict with no parseable findings.

**This is a parser-side fix, not a prompt or calibration-set fix.**
Surface area: `d2p/agents/verifier.py:_parse_result` and adjacent
schema enforcement. Worth chasing before iterating on the calibration
set further — the harness can't get useful category-matching signal
until findings actually round-trip.

### 2. Two productized baselines have a real test-discovery bug.

`flask-clean/tests/test_app.py` does `from app import app`. When the
pre-evidence runner does `pytest -q` from the baseline root, pytest
can't resolve `app` because the baseline isn't installed as a package
and there's no `conftest.py` adding the baseline root to `sys.path`.
Same defect on `python-cli-clean`.

So the verifier's "fail" verdict on these two is NOT a false positive —
it's a true positive on a defect the calibration author seeded by
accident.

**Fix in a follow-up:** add a per-baseline `conftest.py` (sample-project
sized, not the harness conftest) with `sys.path.insert(0, ".")` so the
baseline's tests actually run from its root. After that fix, re-run and
see whether these two switch to `clean_pass`.

`node-cli-clean` was also flagged as FP (verdict `needs_repair`). That
one is less clearly explainable from the report — possibly the verifier
saw `npm test` exit 127 (no `npm` on this dev machine) and concluded
tests aren't actually running. Investigate after the parser fix lands.

### 3. catch_rate=1.00 is encouraging but premature to celebrate.

Because category matching never fires (#1), the catch_rate=1.00 only
proves the verifier returns `needs_repair`/`fail` on broken baselines —
not that it identifies the specific defect. Once the parser is fixed and
`category_match` actually evaluates, the real catch_rate may drop. The
spec §11 target is "needs_repair or fail" verdicts, which is technically
met, but the meaningful signal (right verdict + right category) is
unmeasured.

content triggered larger token output. Not blocking.

## What's next (in order)

1. **Verifier parser fix.** Inspect `_parse_result` and the model's
   actual JSON schema; tighten the parser to extract `findings` /
   `new_finding_categories` correctly across both key spellings. Add a
   parser-only unit test using the captured raw_response as fixture.
2. **Fix the productized baseline test imports.** Per-baseline
   `conftest.py` or restructure to package layout. Re-run harness;
   expect 2 of 3 FPs to flip to `clean_pass`.
3. **Re-run calibration after both fixes** and observe the real
   category-matched catch_rate.
4. **Decide on `node-cli-clean`** based on the new evidence.
5. **Only then** consider scaling to the full §11 set
   (20 broken + 10 productized) or turning `D2P_VERIFY_ENABLED=1` on by
   default.
