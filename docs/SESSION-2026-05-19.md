## What d2p is

A multi-agent loop that turns a **demo repo** into a **product repo**.
No hardcoded "demo type" detectors — the agents read the project, search
the web for mature competitor products in the same domain, and figure the
rest out themselves. Adding a new demo type (audio, blockchain, …) requires
**zero code changes**.

```
Analyzer ── web-search ── extract features + UI + ESSENCE
   │                                    │
   ▼                                    │ (essence/audience are immutable invariants)
Planner ── reads symbol map + key files ─▶ Task list (capped, sorted by priority)
   │
   ▼
Executor[N] ── ===FILE=== or ===PATCH=== (search/replace) ─── sandbox writes
   │
   ▼
QA Agent ── runs corpus → writes failing tests → emits fix-Tasks
                                                     │
                                                     ▼
                                              Fix Executor (uses fix role)
                                                     │
                                                     ▼
                                              (next iter)
```

## Core safety nets (built in priority order across the session)

1. **Sandbox** — every write forced inside target dir.
2. **Destructive-shrink guard** for full-file writes (rejects > 60 % shrink on ≥ 80-line files).
3. **Destructive-shrink guard for PATCH outputs** — same check after applying all SEARCH/REPLACE pairs. *(plugged the `player.py 108→28` leak)*
4. **Syntax check** post-write, language-aware (Python: `ast.parse`; JS: `node --check`).
5. **Self-heal** — on syntax fail, ask the model to fix once with line-numbered file + error.
6. **SEARCH/REPLACE fuzzy fallback** — tolerate leading-whitespace differences; refuse on ambiguity.
7. **Patch SEARCH-miss retry** — one extra LLM round with the actual file + line numbers.
8. **`forbidden_files`** — fix tasks get the test-path in this list so they can't weaken/delete the bug test.
9. **ProjectHealth import probe** — module-level `import X` for every top-level `.py` + every `tests/*.py` + every `tests/d2p_qa/*.py`. Catches *symbol-deletion regressions* that pure syntax check misses.
10. **Baseline-test gate (Fix C)** — runs demo's own `tests/test_smoke.py` before each task; if any was-passing test flips to fail, rollback.
11. **Snapshot/restore** — per-task snapshot inside the write lock so health/baseline regressions roll back precisely the offending task.
12. **`_meta.json` real-time status flip (P3)** — fix's post_check passes → bug status flips to `fixed` immediately, not next iter.
13. **Multi-attempt fix retry (P1-correct)** — `Executor.MAX_FIX_ATTEMPTS=3`; each retry gets `_extract_assertion_summary(output)` to narrow on the failing assertion.

## Speed optimizations (the A+B+C round)

- **A**: Health probe + baseline tests **moved outside the per-file lock**. Now multiple parallel tasks' subprocess probes run concurrently instead of serialised by the write lock.
- **B**: `health_baseline` + `test_baseline` cached **once per iter** at the start of `_run_tasks_parallel`, then shared by every task in the batch. Used to re-probe inside every task lock — 80 % subprocess overhead removed.
- **C**: Default `--parallel` bumped 2 → 4.

Result: 2-iter werewolf-demo: **80 min → 40 min** (50 % faster).

## Language adapters (T11–T15)

The original system was Python-locked. Refactored into `d2p/lang/`:

```
d2p/lang/
  base.py        LanguageAdapter Protocol + NullAdapter
  detect.py      detect_primary_language(sandbox) by file-extension count
  python.py      ast.parse + python -m unittest / pytest (auto-detect)
  javascript.py  node --check + node --test
```

- `ProjectHealth` and `Executor` take an `adapter`; chosen by detect at construction.
- `QAAgent.test_runner_cmd` switches `unittest` ↔ `pytest` based on the test file's own imports. *(P5: fixed the werewolf-demo Fix-C blind spot — `test_smoke.py` is pytest.)*
- **Python picker prefers 3.10+** (Conda envs + Homebrew) over `/usr/bin/python3=3.9` so PEP-604 demos can be probed at all.

## QA loop properties (the "tests grow over time" goal)

- Tests live in `<target>/tests/d2p_qa/`. Each iteration's QA agent runs the entire corpus first (regression sweep), then probes for new bugs informed by `qa_checklist.json` (input_validation / auth / concurrency / resource_lifecycle / error_propagation / state_machine / api_contract / i18n / deployment / observability).
- New bugs are validated by running the freshly-written test: tests that pass on arrival (false positives) are deleted; tests that fail are kept and marked `open` in `_meta.json`.
- Test quality validator rejects truncated outputs (`<300` bytes), missing TestCase, missing `def test_`, and tests with trailing-syntax suggesting truncation.
- `T8: detect_missing_symbol_failures(output)` auto-spawns priority-0 *restore-symbol* tasks when test output is `ImportError: cannot import name 'X' from 'Y'`. (Rarely fires now because T2 health probe stops symbol deletion at the source.)
- `T10 soft throttle`: when `open_bugs >= 12`, feature task count capped to `always_min_features=1` (was: "completely skip features"). Prevents the iter-10 pathology where features-stop-forever made the system produce *less* than a 3-iter run.

1. **Cost/token visibility** — Claude CLI already returns `total_cost_usd`; we discard it. Aggregate into run summary. (~30 min)
2. **End-of-iter "what changed" markdown summary** — auto-generated diff digest at the end of every iter. User asked for this. (~20 min)
3. **Convergence early-stop** — if iter produces 0 features done AND 0 fixes, break before LLM-wasting more iters. (~10 min)

### Medium value
4. **Analyzer periodic re-run** — currently only runs at iter 1. After many iters the project is unrecognisable; the essence/feature list is stale. Re-run every N iters. (~1 hr)

## Lessons from this session

- **Throttling QA is the wrong fix for "find > repair"** — improving fix quality (Sonnet for the fix role) is the right one. Hiding bugs is dishonest.
- **Per-task overhead matters enormously** — moving health probes out of write locks cut wall time in half without touching the LLM at all.
- **Python version assumptions in the probe** break safety nets silently. Always pick a Python that can actually parse modern demos.
- **Destructive guards must run AFTER all writes** — a sequence of small patches can collectively delete a file even though each one passes the per-patch guard.
- **Multi-attempt fix retry with extracted assertion summary** is what moved fix success from 0 → 75 % at the model layer.
- **Don't make the system more clever — make the model better at the role.** Sonnet > Haiku for fix; Opus > Sonnet for Analyzer. Use the cheapest model that's good enough at each role.
